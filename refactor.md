## P1 — Blocks Production Hardening

### 1. No CI/CD Pipeline

**Missing:** `.github/workflows/` directory entirely.  
**Impact:** No automated testing, no lint gate, no build verification on PRs. Broken code ships silently.

**Required files:**
```
.github/
  workflows/
    ci.yml          # lint + test on every PR (all services)
    docker-build.yml  # verify Docker images build on main
  dependabot.yml    # automated dependency security updates
```

---

### 2. No Tests Anywhere

**Missing:** Test directories and frameworks in every service.

| Service | Missing |
|---------|---------|
| `complaint-service/` | `tests/` folder, `pytest`, `conftest.py` |
| `apps/api-server/` | `src/__tests__/`, Jest or Vitest config |
| `apps/dashboard/` | `src/__tests__/`, Vitest + Testing Library setup |

**Required additions per service:**

```
complaint-service/
  tests/
    unit/
      test_ai_classifier.py
      test_policy_engine.py
      test_supervisor_parser.py
    integration/
      test_webhook_flow.py
      test_tickets_api.py
    conftest.py        # shared fixtures, in-memory SQLite setup

apps/api-server/
  src/
    __tests__/
      routes/
        health.test.ts
        dashboard.test.ts

apps/dashboard/
  src/
    __tests__/
      components/
        TicketsTable.test.tsx
```

---

### 3. No Database Migrations

**complaint-service (Python / SQLite):**  
SQLAlchemy creates tables via `Base.metadata.create_all()` on startup. This works for greenfield but has no upgrade path — schema changes on a running database will silently do nothing or crash.

**Required:** Add Alembic.
```
complaint-service/
  alembic/
    versions/          # one file per migration
    env.py
    alembic.ini
```

**apps/api-server (Node / PostgreSQL / Drizzle):**  
`lib/db/drizzle.config.ts` exists but there is no `drizzle/migrations/` directory, meaning `drizzle-kit migrate` has never been run. Schema changes are not versioned.

**Required:**
```
lib/db/
  drizzle/
    migrations/        # generated by drizzle-kit generate
```

## P2 — Enterprise Hygiene

### 4. Non-Standard Top-Level Folder Names

Enterprise monorepos use predictable, well-understood names. Current layout deviates:

| Current | Standard | Reason |
|---------|----------|--------|
| `apps/` | `apps/` | "artifacts" implies build output, not source applications |
| `complaint-service/` at root | `services/complaint/` | Services belong inside a `services/` grouping |
| `scripts/` | `scripts/` ✅ | Fine, but see issue #5 |

**Recommended root structure:**
```
SocietyOps/
  apps/
    dashboard/         # React SPA (moved from apps/dashboard)
    api-server/        # Express API (moved from apps/api-server)
  services/
    complaint/         # Python FastAPI (moved from complaint-service/)
  lib/                 # shared packages (unchanged)
  scripts/
  infra/               # docker-compose, nginx configs, future k8s
  .github/
```

> Migration can be done incrementally — rename `apps/` → `apps/` first since it only affects Docker build context paths in `docker-compose.yml`.

---

### 5. `scripts/` Folder Has Mixed Concerns

Current contents:
```
scripts/
  post-merge.sh        # git hook
  sync-gdrive-rclone.sh
  sync-gdrive-rsync.sh
  src/hello.ts         # TypeScript source inside scripts?
  tsconfig.json
  package.json
```

**Problems:**
- `src/hello.ts` is a stub/test file with no purpose in production.
- Git hooks (`post-merge.sh`) should live in `.githooks/` and be registered via `git config core.hooksPath .githooks`.
- Google Drive sync scripts are ops tooling, not build tooling — they belong in `scripts/ops/` or `infra/`.

**Recommended:**
```
.githooks/
  post-merge            # renamed, no .sh extension for git compatibility
scripts/
  ops/
    sync-gdrive-rclone.sh
    sync-gdrive-rsync.sh
```

### 6. `mockup-sandbox` in Production Artifacts

`apps/mockup-sandbox/` is a Vite component preview sandbox. It:
- Duplicates 70+ UI components from `apps/dashboard/src/components/ui/`
- Has a generated `.generated/mockup-components.ts` file checked into source control
- Is never referenced by docker-compose or any deployment

**Recommended:** Either delete it or move to a top-level `sandbox/` folder explicitly excluded from Docker builds. Add it to `.dockerignore` immediately.

---

### 7. Dead Code — OpenClaw Integration

Two files exist for a legacy WhatsApp gateway that has been superseded by Evolution API:
- `complaint-service/app/routers/openclaw.py`
- `complaint-service/app/services/openclaw_client.py`

These are registered in the FastAPI app and add unused HTTP surface area. **Delete both** and remove the router registration in `main.py`.

---

### 8. Missing Error Handling Middleware

**complaint-service:** FastAPI has no global exception handler. Unhandled exceptions return a raw 500 with a full Python traceback in the response body — a security information leak.

**Required addition in `app/main.py`:**
```python
@app.exception_handler(Exception)
async def global_exception_handler(request, exc):
    logger.exception("Unhandled error", exc_info=exc)
    return JSONResponse(status_code=500, content={"detail": "Internal server error"})
```

**apps/api-server:** Express has no error-handling middleware (the `middlewares/` folder is empty). Unhandled errors propagate to Express's default handler which may leak stack traces.

**Required addition in `src/app.ts`:**
```typescript
// Must be registered after all routes
app.use((err: Error, req: Request, res: Response, _next: NextFunction) => {
  logger.error({ err }, 'Unhandled error');
  res.status(500).json({ error: 'Internal server error' });
});
```

---

### 9. No API Versioning

All API routes are unversioned (`/tickets`, `/api/dashboard`, etc.).  
Any breaking change requires coordinating all consumers simultaneously.

**Recommended prefix:** `/v1/` on both services.
- FastAPI: `app.include_router(tickets_router, prefix="/v1")`
- Express: `router.use('/v1', routes)`

---

### 10. Missing `.editorconfig`

Without an `.editorconfig`, different developers/editors will produce inconsistent indentation, line endings, and trailing whitespace — especially between the Python service (PEP8: 4 spaces) and the TypeScript code (2 spaces).

**Add to root:**
```ini
root = true

[*]
end_of_line = lf
insert_final_newline = true
trim_trailing_whitespace = true

[*.py]
indent_style = space
indent_size = 4

[*.{ts,tsx,js,json,yaml,yml}]
indent_style = space
indent_size = 2
```

---

### 11. Platform Files Polluting the Root

Three Replit-specific files sit at the root:
```
.replit
.replitignore
replit.md
```

These are deployment-platform config, not project config. Move to a `deploy/replit/` folder or add a `.replit/` directory. The `replit.md` content should either be merged into `README.md` or deleted.

## P3 — Polish

### 12. No `CHANGELOG.md`

Conventional Commits + a `CHANGELOG.md` provides traceability for releases. Can be auto-generated with `conventional-changelog-cli` or `release-please`.

---

### 13. `complaint-service/` Has No `config.py`

Settings loaded via `pydantic-settings` should be centralised in a dedicated `app/config.py` (or `app/settings.py`), not scattered across individual service files. This also enables settings injection in tests.

**Recommended:**
```
complaint-service/app/
  config.py        # one Settings class, one place to read env vars
```

---

### 14. No `types/` or `interfaces/` in `api-server`

All TypeScript types are either inline in route files or absent. A `src/types/` directory makes contracts explicit and discoverable.

```
apps/api-server/src/
  types/
    dashboard.ts
    policies.ts
    ai.ts
```

---

### 15. React `pages/` vs `components/` Confusion

`apps/dashboard/src/pages/` contains only `not-found.tsx` while actual pages (`Dashboard.tsx`, `PoliciesPage.tsx`, `ConnectPage.tsx`, `PromptPage.tsx`) live in `components/`. Pages and reusable components should be separated:

```
src/
  pages/
    DashboardPage.tsx     # thin layout wrappers, route targets
    PoliciesPage.tsx
    ConnectPage.tsx
    PromptPage.tsx
    NotFoundPage.tsx
  components/
    dashboard/            # reusable sub-components only
    ui/
```
