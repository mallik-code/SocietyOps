# SocietyOps — Claude Code Guidelines

@.claude/codebase.md

## Project Overview

SocietyOps is a full-stack, AI-powered complaint management platform for residential societies.
Residents submit complaints via WhatsApp groups; the system classifies them with GROQ AI,
creates tickets, replies automatically, and surfaces everything on a React dashboard.

**Monorepo layout (pnpm workspaces)**

```
SocietyOps/
├── complaint-service/       Python 3.12 FastAPI — core complaint engine
├── artifacts/
│   ├── api-server/          Node 22 Express 5 — dashboard API
│   └── dashboard/           React 19 + Vite SPA
├── lib/
│   ├── db/                  Drizzle ORM + PostgreSQL schema
│   ├── api-spec/            OpenAPI spec + Orval codegen config
│   ├── api-client-react/    Generated React Query hooks (DO NOT hand-edit)
│   ├── api-zod/             Generated Zod schemas (DO NOT hand-edit)
│   ├── integrations-openai-ai-server/   Server-side OpenAI client
│   └── integrations-openai-ai-react/    Client-side voice/audio hooks
└── docker-compose.yml       Orchestrates all 5 Docker services
```

## Architecture Principles

### 1. Strict Layer Separation (Python service)

| Layer | Responsibility | Rule |
|-------|---------------|------|
| `routers/` | HTTP binding, input validation, serialization | Never contain business logic |
| `services/` | Business logic, AI calls, external HTTP | Never import from `routers/` |
| `models.py` | ORM schema only | No logic, no imports from services |
| `schemas.py` | Wire-format contracts | Pure Pydantic, no ORM imports |

**Violation example to avoid:** calling `evolution_client.send_message()` directly inside a router — put it in a service.

### 2. Dependency Direction

```
routers → services → models / external clients
schemas ← routers (for validation)
schemas ← services (for response shapes)
```
Circular imports are a hard failure. Services must never import from routers.

### 3. Monorepo Shared Libraries

- `lib/api-client-react` and `lib/api-zod` are **generated by Orval** from `lib/api-spec`.
  Never edit them by hand. Run `pnpm codegen` (from root) to regenerate.
- `lib/db` owns the canonical DB schema. Import `@workspace/db` in api-server; do not duplicate table definitions.
- Cross-service communication inside Docker uses the container name: `http://api:8000`, `http://evolution:8080`.

### 4. Configuration Over Code

All tuneable behaviour lives in env vars (see `.env.example`). Never hard-code:
- API keys, URLs, timeouts
- Policy thresholds (`MIN_CONFIDENCE`, `MAX_MESSAGE_LENGTH`, `ALLOWED_GROUPS`)
- Scheduler timing (`REPORT_CRON_HOUR`, `REPORT_TIMEZONE`)

Policy defaults go in `app/services/policy_engine.py` only as fallbacks when env vars are absent.

### 5. Graceful Degradation

The system must function at reduced capacity when optional dependencies are down:
- **GROQ unavailable** → fall back to keyword classifier (never crash).
- **Evolution API unreachable** → log the error, still create the ticket, skip the WhatsApp reply.
- **READ_ONLY_MODE=true** → process and classify but write nothing; return a dry-run response.

## Coding Standards

### General Rules

You are a production-grade code generator.

Focus on:
- scalability
- maintainability
- reliability

Always include:
- structured logging
- observability hooks
- retry and failure handling

Do not:
- generate demo-level code
- skip non-functional requirements

Ensure:
- API contracts are respected
- code is ready for deployment

### Python (complaint-service)

- **Python 3.12**. Use `from __future__ import annotations` where needed for forward refs.
- **Pydantic v2** everywhere for validation. No raw `dict` for external data.
- **SQLAlchemy 2 async style** is acceptable; sync sessions are fine for current scale.
- **Type annotations are mandatory** for all function signatures. No `Any` unless unavoidable.
- **No mutable default arguments.** Use `field(default_factory=...)` or `None` + guard.
- Keep route handlers thin (< 20 lines). Extract logic into a service.
- Use `async def` for route handlers and services that call external I/O (httpx, AI).
- Never use `print()` for diagnostic output — use the structured logger (`logging.getLogger(__name__)`).
- Exception handling: catch specific exceptions, not bare `except Exception`. Log with `logger.exception()` to preserve tracebacks.

```python
# Good
async def classify(text: str) -> ClassificationResult:
    ...

# Bad
def classify(text):
    ...
```

### TypeScript (api-server + dashboard)

- **Strict mode** is enabled in `tsconfig.base.json`. No `ts-ignore` without a comment explaining why.
- **No `any`** in production paths. Use `unknown` + type guard or Zod parse.
- Generated files (`lib/api-client-react`, `lib/api-zod`) are exempt from linting; ignore with `// eslint-disable-file`.
- API route handlers in Express must end with `res.json(...)` or `next(err)` — never leave the request hanging.
- Use `pino` structured logging everywhere. No `console.log` in production code.
- SSE endpoints (`/api/ai/...`) must call `res.end()` in a `finally` block to close the stream.

### React / Frontend

- Components live in `src/components/`. Page-level compositions in `src/components/<PageName>.tsx`.
- UI primitives from `src/components/ui/` (shadcn/Radix). Never re-implement what Radix provides.
- Data fetching uses **React Query** hooks from `@workspace/api-client-react`. No raw `fetch` in components.
- Forms use **React Hook Form + Zod** resolver. No uncontrolled `useState` for form fields.
- `wouter` for routing — no react-router-dom.
- Keep components pure and free of side effects where possible. Derive state; don't sync it.

## Design Patterns

### Policy Engine (Two-Phase Validation)

The `policy_engine.py` runs two checks. Do not bypass either phase:

```
Phase 1 — Inbound gate (before AI):
  group_filter → sender_filter → keyword_block → length_check

Phase 2 — Post-classification gate (after AI):
  confidence_threshold → read_only_mode → casual_reply_policy
```

New policy rules go in the appropriate phase. Phase 1 is cheap; Phase 2 needs AI output.

### Webhook Idempotency

Evolution API can deliver duplicate events. The `POST /evolution/events` handler must
deduplicate by `message_id` before ticket creation. If deduplication logic is changed,
update `MessageLog` lookups accordingly.

### Supervisor Commands

Supervisor messages follow the pattern `<ticket_id> <action>` (e.g., `42 resolved`).
Parsing lives exclusively in `supervisor_parser.py`. Any new command format must be
added there — not inline in the router.

### Drizzle Schema Changes

All schema changes go through `drizzle-kit generate` + `drizzle-kit migrate`.
Never write raw `ALTER TABLE` SQL for the `lib/db` tables. Raw SQL in `schema.sql`
(complaint-service) is documentation only; truth is in `models.py`.

## Docker & Infrastructure Rules

### Service Names (docker-compose internal DNS)

| Service | Internal host | External port |
|---------|--------------|---------------|
| postgres | `postgres` | — (not exposed) |
| evolution | `evolution` | 8080 |
| api (FastAPI) | `api` | 8000 |
| api-server (Express) | `api-server` | 3001 |
| dashboard (nginx) | `dashboard` | 3000 |

**Never** change internal service names without updating all cross-service URL references.

### Dockerfile Rules

- Multi-stage builds are mandatory for all production images.
- Production stage must not contain dev tools (`npm`, `pip install`, compilers).
- Use `COPY --chown=nonroot:nonroot` and run as a non-root user where the base image supports it.
- Pin image tags (e.g. `python:3.12-slim`, not `python:latest`).
- Add `HEALTHCHECK` to every service image.

### Volume Mounts

- `sqlite_data` → complaint-service only. No other service touches `/app/data`.
- `postgres_data` → Evolution API only.
- `evolution_data` → Evolution API instance sessions (contains QR/auth state).

## Security Rules

- **Never commit** `.env`, secrets, or API keys. `.env.example` provides documented placeholders.
- All env vars that contain secrets must be named with suffix `_KEY`, `_SECRET`, or `_PASSWORD`.
- `CORS_ORIGINS=*` is acceptable for local dev only. For production, restrict to your domain.
- AUTHENTICATION is not implemented (development phase). Before any public exposure, add auth middleware to both the FastAPI backend and the Express api-server.
- Validate all external input with Pydantic (Python) or Zod (TypeScript) **at the boundary** — route handlers. Do not re-validate in services.
- `EVOLUTION_API_KEY` must differ between dev and production.

## Testing Approach

### complaint-service

- Unit test service functions in isolation (mock `evolution_client`, `ai_classifier`).
- Integration tests hit a real SQLite in-memory DB — do not mock the DB layer.
- The policy engine has a `/policy/simulate` endpoint for dry-run testing without sending WhatsApp messages.

### api-server / dashboard

- `pnpm test` from workspace root runs all tests.
- Generated code (`lib/api-client-react`, `lib/api-zod`) does not need tests.
- React component tests should test behaviour, not implementation (use Testing Library, not snapshots).

## Anti-patterns (Do Not Do)

- Do not import `evolution_client` or `ai_classifier` directly in router handlers.
- Do not add `print()` for debugging — use the logger.
- Do not hand-edit `lib/api-client-react/` or `lib/api-zod/` — they are generated.
- Do not share the SQLite volume (`sqlite_data`) with api-server; they use separate databases.
- Do not duplicate table definitions across `lib/db` and `complaint-service/app/models.py` — they own different databases.
- Do not use `POSTGRES_*` env vars in the FastAPI service; it uses SQLite only.
- Do not hard-code `localhost` URLs in Docker service configuration — use container names.
- Do not store session state in the FastAPI process — use the DB; the service may restart.
